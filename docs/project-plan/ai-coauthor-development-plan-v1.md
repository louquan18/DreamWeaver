# DreamWeaver AI 共创开发规划 v1

**版本**: v1.0  
**日期**: 2026-06-18  
**状态**: 规划草案，基于 `docs/product/ai-coauthor-business-design-v1.md`  
**目标**: 将当前“创建小说 + 创建章节 + 初步记忆 + 多 Agent 流转”的 MVP，升级为“Agent 主动编剧、作者担任总编辑”的 AI 修仙小说共创闭环。

---

## 1. 规划目标

当前业务设计已经明确：

- 面向想玩 AI 共创的普通用户。
- 第一版聚焦玄幻 / 修仙题材。
- Agent 主动生成剧情方案，作者负责选择、调整、确认。
- 小说创建阶段生成轻量蓝图。
- 每章先生成 3 个中纲方案，再生成正文。
- 正文确认后提取记忆，作者确认记忆后章节完成。
- 第一版不做卷结构，强制按章节顺序创作。

本规划的目标是把上述业务规则拆成可开发任务，并明确任务间的串行依赖和并行空间。

---

## 2. 总体交付闭环

第一版应打通下面这条闭环：

```text
创建小说设想
  -> Agent 生成轻量小说蓝图
  -> 作者确认蓝图
  -> 创建下一章草稿
  -> Agent 生成 A/B/C 三个中纲方案
  -> 作者选择、混选或调整中纲
  -> Agent 生成约 2000 字正文
  -> Agent 评审并修复 P0 硬冲突
  -> 作者确认正文
  -> Agent 提取记忆变更
  -> 作者确认或编辑记忆
  -> 章节进入 CONFIRMED
```

## 3. 阶段划分

### Phase 0: 现状梳理与接口冻结

**目标**: 明确现有 MVP 能力边界，冻结本轮改造的核心业务接口。

**必须串行原因**: 后续数据库、AI 工作流和前端页面都依赖统一领域模型和接口契约。

| ID | 任务 | 产出 | 依赖 | 并行性 |
| --- | --- | --- | --- | --- |
| P0-T1 | 梳理现有 Story/Chapter/Memory/Generation 实现 | 现状说明与缺口列表 | 无 | 可与 P0-T2 并行 |
| P0-T2 | 梳理现有 LangGraph / Agent 流程 | 当前节点图与状态字段 | 无 | 可与 P0-T1 并行 |
| P0-T3 | 定义 AI 共创领域模型与状态流 | 模型与状态设计 | P0-T1, P0-T2 | 串行 |
| P0-T4 | 定义前后端 API 契约草案 | OpenAPI 或 Markdown API 草案 | P0-T3 | 串行 |

**完成标准**:

- 明确哪些已有表和接口复用，哪些新增。
- 明确 `DRAFT` / `CONFIRMED` 主状态和内部任务状态。
- 明确小说蓝图、章节中纲、正文草稿、记忆变更集的接口结构。

| P1-T3 | 设计 `MemoryChangeSet` | 待确认记忆变更模型 | P0-T3 | 可与 P1-T1/P1-T2 并行 |

| P1-T5 | 调整 Chapter 状态与顺序约束 | `DRAFT` / `CONFIRMED` 规则 | P1-T1~P1-T4 | 串行 |

**目标**: 创建章节后不直接生成正文，而是先生成 A/B/C 三个中纲方案。

| P3-T3 | 实现上下文加载适配 | 蓝图、主线、历史章节、记忆、伏笔 | P1-T6 | 可与 P3-T2 并行 |
| P3-T4 | 实现三方案生成 Agent | `generate_outline_options` | P3-T1, P3-T2, P3-T3 | 串行 |
| P3-T5 | 实现 C 方案伏笔兜底逻辑 | 回收 -> 强化 -> 埋设 | P1-T4, P3-T4 | 串行 |
| P3-T6 | 实现作者选择/混选/调整 API | 最终中纲确认 | P1-T2, P3-T4 | 串行 |
| P3-T7 | 前端三方案卡片与解释展示 | A/B/C 对比 UI | P0-T4, P3-T1 | 可与 P3-T4 并行 |

**完成标准**:

- 中纲确认前不能进入正文生成。

| ID | 任务 | 产出 | 依赖 | 并行性 |

| P4-T5 | 实现评审 Agent | `review_quality` | P4-T2, P4-T4 | 串行 |
| P4-T6 | 实现一致性 Agent | `check_consistency` | P4-T3, P4-T4 | 串行 |
| P4-T7 | 实现 P0 自动修复流 | 局部/整章修复 | P4-T5, P4-T6 | 串行 |

## 4. 串行关键路径

以下任务构成主关键路径，必须按顺序推进：

```text
P0-T1/T2 现状梳理
  -> P0-T3 领域模型与状态流
  -> P0-T4 API 契约
  -> P1 数据模型与迁移
  -> P2 蓝图生成与确认
  -> P3 三方案中纲与确认
  -> P4 正文生成、评审、修复与确认
  -> P5 记忆提取、确认与章节冻结
  -> P6 工作台整合
  -> P7 端到端验收
```

关键路径上的任何延误都会影响完整闭环交付。

## 5. 可并行任务分组

### 5.1 第一批并行：现状梳理

可以并行：

- `P0-T1` 梳理现有 Story/Chapter/Memory/Generation。
- `P0-T2` 梳理现有 LangGraph / Agent 流程。

必须等二者完成后：

- `P0-T3` 领域模型与状态流。
- `P0-T4` API 契约。

### 5.2 第二批并行：数据模型拆分设计

在 `P0-T3` 后可以并行：

- `P1-T1` NovelBlueprint。
- `P1-T2` ChapterOutlineOption / ChapterOutline。
- `P1-T3` MemoryChangeSet。
- `P1-T4` Foreshadow 生命周期。

必须等这些完成后：

- `P1-T5` Chapter 状态与顺序约束。
- `P1-T6` 迁移脚本与 Repository。

### 5.3 第三批并行：AI Prompt 与前端壳

在 API 契约初稿稳定后，可以并行：

- 蓝图 Prompt 与蓝图前端表单。
- 中纲 Schema/Prompt 与三方案卡片 UI。
- 评审报告结构与正文评审展示 UI。
- 记忆变更 Schema 与记忆确认面板 UI。

注意：这些可以并行做“壳”和契约，但最终联调必须等待对应后端能力完成。

### 5.4 第四批并行：Agent 子能力

在数据模型完成后，可以部分并行：

- 蓝图生成 Agent。
- 三方案生成 Prompt。
- 上下文加载适配。
- 评审报告结构。
- 一致性检查规则。
- 记忆提取 Prompt。

必须串行接入：

```text
确认中纲 -> 正文生成 -> 评审/一致性检查 -> 修复 -> 正文确认 -> 记忆提取 -> 记忆确认 -> 章节冻结
```

### 5.5 第五批并行：测试随开发推进

可以持续并行：

- 数据层单元测试。
- Agent Prompt 输出结构测试。
- API Service 测试。
- 前端组件测试。

必须最后串行：

- 蓝图闭环 E2E。
- 单章闭环 E2E。
- 连续两章顺序创作 E2E。

## 7. 建议里程碑

### Milestone A: 业务底座完成

包含：

- 领域模型与 API 契约。
- 数据模型与迁移。
- 章节状态与顺序创作约束。

验收：

- 能创建小说蓝图记录。
- 能创建章节草稿。
- 能阻止未完成上一章时创建下一章正式生成任务。

### Milestone B: 小说蓝图闭环完成

包含：

- 设想输入。
- 蓝图生成。
- 表单编辑。
- 蓝图确认。

验收：

- 用户输入一句话后，系统生成并保存完整轻量蓝图。

### Milestone C: 章节中纲闭环完成

包含：

- A/B/C 三方案生成。
- 方案解释。
- 作者选择/混选/调整。
- 最终中纲确认。

验收：

- 中纲未确认不能生成正文。
- C 方案符合伏笔兜底规则。

### Milestone D: 正文生成与评审闭环完成

包含：

- 基于确认中纲生成正文。
- 一致性检查。
- 质量评审。
- P0 自动修复。
- 正文确认。

验收：

- P0 未修复时不能确认正文。
- P1/P2 可以由作者忽略。

### Milestone E: 记忆确认与章节冻结完成

包含：

- 记忆提取。
- 记忆冲突检测。
- 作者编辑确认。
- 写入长期记忆。
- 章节 `CONFIRMED`。

验收：

- 记忆未确认章节不能完成。
- 章节确认后不可修改。
- 下一章只能在上一章确认后进入正式创作。

### Milestone F: 工作台与端到端验收完成

包含：

- 前端工作台整合。
- 完整 E2E 测试。
- 样例小说验收。
- 文档更新。

验收：

- 从一句设想到连续完成 2 章的全流程可跑通。

---

## 8. 推荐开发顺序

如果只有一个主开发流，推荐按下面顺序：

1. 现状梳理与接口冻结。
2. 数据模型与迁移。
3. 小说蓝图生成和确认。
4. 三方案中纲生成和确认。
5. 正文生成接入确认中纲。
6. 评审与 P0 修复。
7. 记忆提取与确认。
8. 章节冻结和下一章顺序约束。
9. 前端工作台整合。
10. E2E 验收。

如果有多人并行，推荐分三条线：

| 线 | 负责内容 | 可并行起点 | 汇合点 |
| --- | --- | --- | --- |
| 后端业务线 | 数据模型、状态机、API、确认动作 | P0-T3 后 | P2/P3/P5 联调 |
| AI 工作流线 | 蓝图、中纲、正文、评审、记忆 Agent | P0-T3 后 | P3/P4/P5 联调 |
| 前端工作台线 | 蓝图表单、三方案卡片、正文确认、记忆确认 | P0-T4 后 | P6 端到端整合 |

## 9. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 领域模型未冻结就开工 | 后续 API 和前端反复返工 | 先完成 P0-T3/P0-T4 |
| 三方案输出结构不稳定 | 前端难展示，后端难保存 | 强制 JSON Schema 和结构化解析 |
| Agent 生成补丁越权写入世界观 | 故事设定漂移 | 世界观补丁必须 pending，作者确认后生效 |
| 记忆提取质量不稳定 | 后续章节上下文污染 | 记忆确认面板必须允许编辑和删除 |
| P0/P1/P2 分级不清 | 作者不知道哪些必须修 | 固定评审报告结构和阻塞规则 |
| 顺序创作约束遗漏 | 下一章基于不完整记忆生成 | 后端 Service 层强制校验上一章状态 |
| 正文确认后不可修改导致体验挫败 | 用户误确认后无法挽回 | 确认前提供明确提示，允许确认前重写 |

## 10. 第一轮可执行任务建议

建议下一轮实际开发先做这些任务：

1. `P0-T1` 梳理现有 Story/Chapter/Memory/Generation 实现。
2. `P0-T2` 梳理现有 LangGraph / Agent 流程。
3. `P0-T3` 输出领域模型与状态流最终版。
4. `P0-T4` 输出 API 契约草案。
5. `P1-T1` 到 `P1-T4` 并行设计四类数据模型。
6. `P1-T5` 定义 Chapter 状态和顺序创作约束。

完成这些后，再进入蓝图生成和章节中纲生成开发，会稳很多。
